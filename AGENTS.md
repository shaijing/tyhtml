# AGENTS.md

Project guidance for AI coding agents working on **TyHtml** — a native
Node.js addon (Rust + [napi-rs](https://napi.rs)) that compiles
[Typst](https://typst.app) `.typ` files to HTML and extracts metadata
via the `<meta>` label.

If you are Claude Code, you were loaded here via `@AGENTS.md` from
[`CLAUDE.md`](./CLAUDE.md). Everything below applies to you.

## 3. Architecture

`src/lib.rs` defines the entire Rust side. Key pieces:

- **`BridgeWorld`** — implements `typst::World`. Wraps `Library` (built with `Feature::Html`), a `FontStore` (system fonts + any extra `fontPaths`), and a `FileStore` backed by `BridgeFiles`. Owns `now: Time` so `today()` works.
- **`BridgeFiles`** — `FileLoader` that resolves project files from a real `FsRoot`, plus package files from `~/Library/Application Support/typst/packages` (macOS) / `%LOCALAPPDATA%\typst\packages` (Windows) / `~/.local/share/typst/packages` (Linux).
- **`run_compile`** — pure function: build the world → `typst::compile::<HtmlDocument>` → `typst_html::html` → optional `extract_body` → optional `extract_metadata`. Returns `Result<CompileResult>`.
- **`extract_metadata`** — queries the introspector for a `<meta>` label, reads its `value` field, recursively converts `typst::Value` → `serde_json::Value`, serializes to a JSON string.
- **`extract_body`** — naive string search for `<body…>` and `</body>`; not regex, not a real parser.
- **JS entry points**:
  - `compile_typst` (async, `#[napi]`) — wraps `compile_typst_impl`, which moves blocking work onto `napi::tokio::task::spawn_blocking` so the event loop stays free.
  - `compile_typst_sync` (sync, `#[napi]`) — calls `run_compile` directly on the calling thread.
  - Both share the option-flattening preamble; `compile_typst_impl` was extracted from the original `compile_typst` body to keep the two entry points in sync.

### Mental model

```
JS thread                 worker thread                 typst
─────────                 ─────────────                 ──────
compileTypst ──┐
               │ spawn_blocking ──► run_compile ──► BridgeWorld::new
               │                                    │
compileTypstSync ───────────────► run_compile ──► typst::compile::<HtmlDocument>
                                                    ↓
                                              typst_html::html
                                                    ↓
                                       (extract_body / extract_metadata)
                                                    ↓
                                              CompileResult
```

## 4. Platform matrix

`napi.targets` in `package.json` is the source of truth. Currently:

| Triple | npm sub-package |
|---|---|
| `x86_64-pc-windows-msvc` | `@isomtop/tyhtml-win32-x64-msvc` |
| `x86_64-unknown-linux-gnu` | `@isomtop/tyhtml-linux-x64-gnu` |
| `aarch64-apple-darwin` | `@isomtop/tyhtml-darwin-arm64` |
| `x86_64-apple-darwin` | `@isomtop/tyhtml-darwin-x64` |

**Cross-build toolchain notes:**

- **Linux x64** — `@napi-rs/cross-toolchain` pulls in `zig` ≥ 0.13 and drives `cargo-zigbuild`. Works from any host.
- **Darwin** — needs an Apple SDK. Two options:
  1. Build on macOS directly (`npm run build` picks up the host triple).
  2. Cross-build via `osxcross` — `@napi-rs/cross-toolchain` knows how to wire it; expect to set `SDKROOT` / `OSXCROSS_ROOT` per its docs.
- **Windows x64** — MSVC. `napi build` works on a Windows host out of the box; cross-building from Linux is not currently configured.

When you add a new target: edit `napi.targets`, add a matching `optionalDependencies` entry, add a `build:<triple-short>` script following the existing pattern, and update the platform table in `README.md`.

## 5. Build, test, publish

All commands assume Bun on `PATH`. Replace `bun` with `npm` / `pnpm` / `yarn` if needed — the scripts are standard `napi` invocations.

```bash
# Install JS deps (uses bun.lock)
bun install

# Build the host binary (index.js + index.d.ts + .node land at repo root)
bun run build

# Build every supported target (host + Linux x64 + Darwin arm64 + Darwin x64)
bun run build:all

# Run the smoke test
bun tests/test.ts

# Type-check (tsconfig has noEmit: true)
bunx tsc --noEmit

# Scaffold npm/<triple>/ sub-package directories (gitignored)
bun run create-npm-dirs

# Stage binaries into npm/<triple>/ for publish
bun run artifacts

# One-shot: build → scaffold → stage (what `npm publish` runs via prepublishOnly)
bun run prepublishOnly
```

### Publishing flow (manual)

```bash
# 1. Ensure you are logged into the @isomtop npm account
npm login --registry=https://registry.npmjs.org/

# 2. Run prepublishOnly on every supported target first
bun run prepublishOnly

# 3. Publish root + every per-triple sub-package
npx napi pre-publish
```

The per-triple packages are versioned in lockstep with the root (`0.1.0` today) — bump all of them together when cutting a release.

## 6. Editing the public API

Anything that changes the JS surface lives in `src/lib.rs`. After editing:

1. Re-run `bun run build` — `napi-derive` regenerates `index.d.ts` (gitignored, but it's what consumers see).
2. Update `tests/test.ts` if a behavior changed.
3. Update `README.md` examples to match.
4. Bump the version in `package.json` (and every per-triple sub-package's `package.json` once `napi create-npm-dirs` regenerates them).

**Naming conventions** between Rust and JS (napi handles the conversion, but be consistent):

| Rust | JS |
|---|---|
| `compile_typst` | `compileTypst` |
| `compile_typst_sync` | `compileTypstSync` |
| `body_only` | `bodyOnly` |
| `no_metadata` | `noMetadata` |
| `metadata_label` | `metadataLabel` |
| `font_paths` | `fontPaths` |

`CompileOptions` is `Option<…>` everywhere with `unwrap_or`/`unwrap_or_else` defaults applied in Rust — never force the JS caller to send a value.

## 7. Style and conventions

- **Rust** — match the surrounding code: 4-space indent, `///` doc comments on every `#[napi]`-exported item, `anyhow::Result` internally, `napi::Result` at the JS boundary. Use `format!("{e:#}")` (alternate display) when stringifying anyhow errors so the chain is preserved.
- **`#[napi]` objects** — field order in the struct is what JS sees. Adding a field is non-breaking; renaming or removing is breaking.
- **Error messages** — keep them user-readable. Compilation errors get prefixed with `Compilation failed:\n` so the caller can tell typst errors from napi errors.
- **Tests** — `tests/test.ts` is currently a smoke test that compiles `tests/fixtures/hello.typ` and prints HTML + metadata. Add fixtures rather than asserting against hardcoded HTML — Typst's output is version-pinned and changes between minor versions.
- **Comments** — section dividers like `// ── JS-facing types ────────────────────────────────` mirror the file's actual structure; preserve them when adding new sections.

## 8. Common pitfalls

- **`BridgeWorld::new` calls `canonicalize()` on the input path** — if the caller passes a relative path or a symlink that doesn't resolve, the function errors with `"Cannot resolve input file path"`. Surface this clearly; do not silently fall back.
- **`typst-kit` font discovery is platform-specific** — `typst_kit::fonts::system()` returns different sets on each OS. If a user reports "missing font" bugs, ask which platform first.
- **`extract_body` is naive string search** — it does not understand attributes on `<body>` (e.g. `<body class="...">` is handled, but malformed nesting is not). Don't extend it; replace it with a real HTML parser if a stronger guarantee is ever needed.
- **Worker thread blocking** — `compile_typst` already moves work off the JS thread via `spawn_blocking`. Do not add a sync path that bypasses this without an explicit reason; the sync entrypoint (`compile_typst_sync`) is the only sanctioned blocking variant.
- **Optional dependencies** — adding a new platform target means publishing a new npm sub-package first, otherwise `bun install` on that platform silently skips the binary and the import throws at runtime.
- **`packageManager` field** — keep it pinned (`bun@1.4.0` today). Corepack and other tooling will refuse to install if the lockfile was produced by a different version.
